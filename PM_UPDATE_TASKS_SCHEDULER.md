# 任务反馈重设计 + 定时任务设置 产品需求文档

## 1. 目标

本次需求解决两个核心问题：

1. 当前任务执行反馈不够清晰，用户难以快速判断任务是否未开始、已启动、正在执行、执行成功或执行失败。
2. 当前任务触发方式偏单一，缺少清晰、可配置的定时任务设置能力，无法满足“手动执行 + 定时执行”并存的使用场景。

本次目标是：

- 建立统一、可感知、低歧义的任务执行状态反馈体系。
- 让用户在任务列表和任务详情中，都能快速理解当前状态与最近一次执行结果。
- 提供简洁可配置的定时任务设置区块，支持常见调度配置。
- 明确定义手动执行与定时执行的关系，避免互相覆盖、冲突或状态混乱。
- 为前后端联调提供明确的状态枚举、字段要求与接口约束。

---

## 2. 任务执行反馈重设计要求（空闲/启动/运行中/成功/失败）

### 2.1 状态定义

任务执行反馈统一收敛为以下 5 个主状态：

1. **空闲**
   - 定义：当前没有执行中的任务，且最近一次执行未在当前时刻产生进行中反馈。
   - 典型场景：
     - 任务从未执行过
     - 上一次执行已结束，当前未触发
     - 已配置定时任务，但尚未到下一次执行时间
   - 页面文案建议：
     - `空闲`
     - 辅助信息：`未在执行中` / `等待下次触发`

2. **启动**
   - 定义：用户手动点击执行或系统命中定时触发后，任务已受理，但实际执行流程尚未进入稳定运行阶段。
   - 目标：用于承接“已响应”与“运行中”之间的短暂阶段，减少用户误判为没点上或没触发。
   - 页面文案建议：
     - `启动中`
     - 辅助信息：`任务已触发，正在准备执行`

3. **运行中**
   - 定义：任务已进入实际执行阶段。
   - 典型特征：有开始时间，可选进度信息，可持续刷新运行日志或阶段描述。
   - 页面文案建议：
     - `运行中`
     - 辅助信息：`正在执行任务`

4. **成功**
   - 定义：本次任务执行完成，结果符合成功结束条件。
   - 页面文案建议：
     - `执行成功`
     - 辅助信息：`于 XX:XX 完成`

5. **失败**
   - 定义：本次任务执行结束，但未达到成功条件，包括执行报错、超时、被系统中断、依赖异常等。
   - 页面文案建议：
     - `执行失败`
     - 辅助信息：`于 XX:XX 失败` + 可查看失败原因

### 2.2 状态流转要求

标准状态流转如下：

- 手动触发：`空闲/成功/失败 -> 启动 -> 运行中 -> 成功/失败`
- 定时触发：`空闲/成功/失败 -> 启动 -> 运行中 -> 成功/失败`

补充要求：

- `启动` 必须是可见状态，即使持续时间很短，也要保证用户在 UI 上可感知。
- 若后端无法长时间停留在 `启动`，前端需保证最短展示时长，建议不少于 600ms，避免状态闪烁不可见。
- `运行中` 结束后只能落到 `成功` 或 `失败`，不能直接回到 `空闲` 而丢失结果反馈。
- `成功` / `失败` 属于“最近一次执行结果态”，保留到下一次新执行开始前。
- 当新一轮执行开始时，最近一次结果态应被新的 `启动/运行中` 覆盖展示，但历史结果仍需可在记录区查看。

### 2.3 列表页反馈要求

任务列表中每个任务至少展示：

- 当前主状态
- 最近一次执行时间
- 最近一次执行结果摘要
- 是否已开启定时任务
- 下一次计划执行时间（若存在）

列表页目标：让用户不进入详情也能快速判断：

- 这个任务现在是不是在跑
- 最近一次是成功还是失败
- 是否配置了定时执行
- 下次什么时候自动跑

### 2.4 详情页反馈要求

任务详情页至少展示：

- 当前状态卡
- 触发来源（手动 / 定时）
- 本次执行开始时间
- 本次执行耗时（运行中可实时增长，结束后显示最终值）
- 最近一次执行结果
- 失败原因摘要（失败时）
- 执行记录入口

详情页目标：让用户理解“这次怎么触发、当前走到哪一步、结果如何、失败在哪”。

## 3. 动效与视觉要求（按钮、状态卡、进度条、Toast/完成反馈）

### 3.1 按钮设计要求

#### 手动执行按钮

- 默认状态：可点击，文案为 `立即执行`
- 启动状态：按钮进入 loading 态，文案为 `启动中`
- 运行中状态：按钮禁用，文案为 `执行中`
- 成功后：按钮恢复可点击；不要求按钮常驻成功态，避免与状态卡重复表达
- 失败后：按钮恢复可点击

#### 定时任务开关/设置按钮

- 需要与手动执行按钮视觉区分，避免用户误以为“设置定时”会立即执行任务。
- 推荐形态：
  - 一个 `定时任务` 设置入口按钮
  - 一个启用/停用开关

### 3.2 状态卡要求

状态卡是任务反馈的核心视觉容器，必须固定展示在任务详情关键区域。

状态卡至少包含：

- 状态标签：空闲 / 启动中 / 运行中 / 执行成功 / 执行失败
- 状态颜色
- 状态说明文案
- 最近更新时间
- 触发来源
- 若有下一次调度，展示下一次执行时间

推荐颜色语义：

- 空闲：中性灰
- 启动：品牌主色或浅蓝
- 运行中：品牌主色/蓝色动态态
- 成功：绿色
- 失败：红色

### 3.3 进度条要求

- `运行中` 状态下可显示进度条。
- 若后端可提供真实进度百分比，则展示确定型进度条。
- 若后端暂时无法提供真实进度，则展示不确定型进度条/条纹动画，用于表达“任务正在执行”。
- `启动` 阶段不强制显示百分比，但可有轻量 loading 动效。
- `成功` / `失败` 后，进度条停止更新，不再继续动画。

### 3.4 动效原则

- 动效目标是增强状态感知，不是制造干扰。
- 所有状态切换应平滑过渡，避免突兀闪烁。
- `启动 -> 运行中`、`运行中 -> 成功/失败` 应有清晰但克制的过渡动画。
- 成功反馈应偏轻量，失败反馈应偏明确。
- 页面切换或刷新后，状态展示必须以真实状态恢复，不能只依赖本地临时动画状态。

### 3.5 Toast / 完成反馈要求

#### 触发后反馈

- 用户手动点击执行后，应立即出现 Toast：`任务已启动`
- 若触发失败（例如请求失败、参数缺失、权限不足），应立即 Toast 明确报错原因

#### 完成后反馈

- 执行成功时：
  - Toast：`任务执行成功`
  - 同时状态卡切换到成功态
- 执行失败时：
  - Toast：`任务执行失败`
  - 同时状态卡切换到失败态，并提供失败原因入口/摘要

#### 反馈约束

- 不使用过度打扰式弹窗作为默认完成反馈。
- Toast 只做短反馈，详细信息统一进入状态卡和执行记录。
- 若任务在后台长时间运行，完成反馈应在用户当前页面内可感知；如产品已有消息中心，可同步进入消息中心，但不是本次必做。

## 4. 定时任务设置区块设计

### 4.1 区块目标

定时任务设置区块用于让用户完成以下操作：

- 开启/关闭当前任务的自动执行
- 配置执行周期
- 查看下一次执行时间
- 理解定时任务是否生效
- 在不打断手动执行能力的前提下，管理自动调度

### 4.2 区块结构

定时任务设置区块至少包含以下内容：

1. **总开关**
   - 开启定时执行
   - 关闭定时���行

2. **调度方式选择**
   - 本期建议支持一种主表达，避免复杂度失控：
     - 方案 A：Cron 表达式输入 + 示例提示
     - 方案 B：可视化频率配置（推荐）

本期推荐主方案：**可视化频率配置优先，Cron 表达式作为高级配置能力可后置或折叠。**

3. **基础配置项**
   - 执行频率：每天 / 每周 / 自定义
   - 执行时间：时:分
   - 若为每周：选择星期几
   - 时区说明（至少展示系统使用时区）

4. **结果展示区**
   - 当前是否已启用定时任务
   - 当前调度规则摘要
   - 下一次执行时间
   - 最近一次定时执行时间

5. **保存操作区**
   - `保存设置`
   - 保存成功/失败反馈

### 4.3 推荐交互

- 当开关关闭时：
  - 配置区可折叠或置灰
  - 展示文案：`当前未启用定时执行`

- 当开关开启但未保存完整配置时：
  - 明确提示：`请完成执行频率与时间设置`
  - 禁止展示“已生效”误导信息

- 保存成功后：
  - 明确展示：`定时任务已生效`
  - 展示规则摘要与下一次执行时间

- 保存失败后：
  - 明确提示失败原因
  - 保留用户已填写内容，避免重新填写

### 4.4 范围控制

本期必须支持：

- 开启/关闭定时任务
- 至少一种稳定可用的周期配置方式
- 展示下一次执行时间
- 与手动执行并存

本期不强制支持：

- 多个调度规则并存
- 节假日跳过
- 按月份/复杂日历规则
- 依赖链式任务编排
- 极其复杂的 Cron 可视化解释器

## 5. 手动执行与定时执行并存规则

### 5.1 并存原则

手动执行与定时执行必须并存，二者目标不同：

- 手动执行：用于用户即时触发
- 定时执行：用于系统按计划自动触发

定时任务的存在不能替代手动执行入口；手动执行入口也不能覆盖定时配置状态。

### 5.2 触发来源标记

每一次执行都必须带有明确触发来源：

- `manual`：手动触发
- `schedule`：定时触发

前端需要在状态卡和执行记录中展示来源，避免用户误解“这次是我点的还是系统自己跑的”。

### 5.3 冲突处理规则

#### 同一任务已有执行中的情况

当任务处于 `启动` 或 `运行中` 时：

- 不允许再次发起新的并发执行（本期默认单任务单实例）
- 若用户点击手动执行，应提示：`任务正在执行中，请稍后再试`
- 若此时命中定时触发，也不应创建第二个并发实例

#### 定时触发撞上运行中的处理

本期默认规则：**跳过本次定时触发，并记录一次“跳过原因：任务执行中”。**

原因：

- 规则简单，用户容易理解
- 避免同一任务并发导致结果污染
- 避免引入排队补跑机制的额外复杂度

本期不做：

- 定时触发排队等待
- 前一轮完成后自动补跑一次
- 多实例并行执行

### 5.4 状态展示规则

- 若当前是手动执行触发，状态卡展示：`手动触发`
- 若当前是定时执行触发，状态卡展示：`定时触发`
- 若最近一次执行成功/失败，也应保留对应来源信息
- 下一次计划执行时间仅由定时配置决定，不受最近一次手动执行影响

### 5.5 用户预期管理

需要明确告知用户：

- 手动执行不会关闭定时任务
- 修改定时设置不会自动立即执行一次任务，除非用户主动点击手动执行
- 当任务正在运行时，新的手动或定时触发不会并发执行

## 6. 验收标准

### 6.1 状态反馈验收

1. 任务存在统一的 5 个主状态：空闲、启动、运行中、成功、失败。
2. 用户点击手动执行后，界面能立即看到 `启动中` 或等价已受理反馈。
3. 任务进入执行后，界面能明确展示 `运行中` 状态。
4. 任务完成后，界面能明确展示成功或失败，且不会直接丢回空闲态。
5. 列表页和详情页都能展示最近一次执行结果。
6. 失败状态下可查看失败原因摘要。

### 6.2 动效与视觉验收

1. 手动执行按钮在启动/运行阶段有明确状态变化。
2. 状态卡固定存在，且颜色、文案、状态标签一致。
3. 运行中存在明确的动态反馈（真实进度条或不确定型进度条）。
4. 手动触发后有即时 Toast 反馈。
5. 成功/失败后有对应 Toast，且状态卡同步更新。
6. 页面刷新后，状态仍能基于真实数据正确恢复。

### 6.3 定时任务设置验收

1. 用户可以开启和关闭定时任务。
2. 用户可以完成至少一种周期配置并保存成功。
3. 保存成功后，界面可展示当前调度规则摘要。
4. 保存成功后，界面可展示下一次执行时间。
5. 保存失败时，界面保留已填写配置并给出错误提示。

### 6.4 并存规则验收

1. 定时任务开启后，手动执行入口仍然可见且可用（非运行中场景）。
2. 手动执行不会清空或关闭已有定时配置。
3. 当任务运行中再次手动触发时，系统会阻止重复执行并给出提示。
4. 当任务运行中命中定时触发时，系统跳过本次触发并记录原因。
5. 执行记录中能区分手动触发与定时触发。

## 7. 开发接口要求

### 7.1 状态枚举要求

前后端统一任务状态枚举，至少包括：

- `idle`
- `starting`
- `running`
- `success`
- `failed`

说明：

- 前端展示文案可为中文
- 接口枚举统一使用英文，避免歧义

### 7.2 任务详情接口字段要求

任务详情接口至少返回以下字段：

- `taskId`
- `taskName`
- `status`：任务当前主状态
- `triggerSource`：`manual` / `schedule` / `null`
- `lastRunAt`：最近一次执行开始时间
- `lastFinishAt`：最近一次执行结束时间
- `durationMs`：最近一次执行耗时
- `lastResult`：`success` / `failed` / `null`
- `lastErrorMessage`：最近一次失败原因摘要
- `scheduleEnabled`：是否启用定时任务
- `scheduleConfig`：定时配置对象
- `nextRunAt`：下一次计划执行时间
- `updatedAt`：状态更新时间

### 7.3 定时配置对象要求

`scheduleConfig` 至少包含：

- `type`：如 `daily` / `weekly` / `custom`
- `time`：执行时间，例如 `09:30`
- `weekdays`：每周执行时的星期数组
- `timezone`：时区
- `expression`：若支持高级表达，可返回原始表达式
- `summary`：用于前端直接展示的规则摘要

### 7.4 执行记录接口要求

执行记录至少返回：

- `runId`
- `taskId`
- `triggerSource`
- `status`
- `startedAt`
- `finishedAt`
- `durationMs`
- `errorMessage`
- `skipReason`（用于定时触发被跳过场景）

### 7.5 手动执行接口要求

手动执行接口要求：

- 调用成功后，立即返回已受理结果，不应要求前端长时间等待直到任务完成。
- 返回内容至少包括：
  - `accepted: true/false`
  - `runId`（若已创建）
  - `status`（初始应为 `starting` 或等价态）
  - `message`

若任务已在执行中：

- 返回明确错误语义，例如：
  - `accepted: false`
  - `code: TASK_ALREADY_RUNNING`
  - `message: 任务正在执行中`

### 7.6 定时任务保存接口要求

定时任务保存接口要求：

- 支持保存启用状态与调度配置
- 保存成功后返回：
  - `scheduleEnabled`
  - `scheduleConfig`
  - `nextRunAt`
  - `message`

- 保存失败时返回明确错误码与错误信息，便于前端直出提示

### 7.7 状态刷新要求

前端需要有真实状态同步机制，至少支持以下之一：

- 轮询任务详情/执行状态接口
- WebSocket / SSE 实时推送

要求：

- 用户触发执行后，状态能从 `starting` 更新到 `running`，再更新到 `success/failed`
- 页面刷新后可恢复真实状态
- 不依赖纯前端本地假状态完成整套反馈

## 附：本期产品建议结论

本期推荐以“**状态反馈清晰化 + 定时配置最小闭环**”为目标收口，不扩展到复杂调度系统。

优先保证：

- 五态反馈清晰
- 手动执行即时可感知
- 定时设置可开启、可保存、可看到下次执行时间
- 手动与定时不冲突
- 接口字段可支撑前端稳定展示

不在本期扩展复杂编排、并发补跑、多规则调度等能力。
